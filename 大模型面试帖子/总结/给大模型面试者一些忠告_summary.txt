给大模型面试者一些忠告
我这几年带了不少校招生，也面试过很多想转到大模型方向的同学。最近又有不少人问我：大模型到底应该做算法还是应用？今天我就以一个已经在行业里站稳、工作几年、现在算是“老学长”的身份，跟你们聊聊我看到的真实情况。不绕弯子，也不贩卖焦虑，只说职场里的真心话。
	
01.大模型里的“算法”，很多人理解错了
很多同学一听算法，就以为一定是发论文、改结构、搞前沿研究。其实不是的。
行业里真正大量存在的，是“工程向算法”——
本质上是让模型跑得更快、更便宜、更稳、更能落地。
你会做这些：
- 模型微调
- 推理加速
- 模型对齐
- Serving 部署
- 成本优化
公司就会愿意花钱留你。
这部分岗位，竞争没那么夸张，价值却非常高。
	
02. 为什么我建议新人往“算法工程”方向靠
一句话：离业务远一点，离核心技术近一点，你的价值就稳一点。
现在做纯 Agent 应用的人真的太多了：
调 API、写 prompt、换框架……
门槛低、上手快，但天花板也明显。
而能把模型效果和性能做出来的人可能不到 20%。
比如：
- 你能把 RAG 命中率从 60% 拉到 85%
- 你能把推理成本从 1/1 砍到 1/3
- 你能让一个任务从 3 秒提速到 0.8 秒
这类人走到哪都吃香，薪资和发展轨迹就是不一样。
纯应用岗开发但一点原理不懂不是没前途，但有瓶颈
我不是贬低应用岗，我带的新人里也有做应用的，前两年确实顺风顺水。
但应用岗有个明显问题：
换一家公司，你几乎要重新适配业务，从零再做。
很难形成可迁移的硬实力。
而算法工程的能力积累是跨业务线通用的。
你会优化、会微调、会部署，不管去哪家公司，都是立刻能贡献价值的那类人。
	
给还在学校或刚毕业的同学，一条实际可走的路线
这是我带新人时的标准路径，你们完全可以照着来。
第一阶段：应用能力打底
- prompt
- LangChain / LlamaIndex
- 基础 RAG 架构
先把这些搞懂，你至少能快速做出一个能上线的 demo。
第二阶段：往下一层理解原理
不用写论文，但要能讲清楚：
- Transformer
- Attention
- KV Cache
- 位置编码
这些东西说不明白，面试深点的岗位就容易卡。
第三阶段：开始形成技术壁垒
选择性深入：
- QLoRA / 全参微调
- DPO / ORPO / RLHF
- INT4 / IN...
